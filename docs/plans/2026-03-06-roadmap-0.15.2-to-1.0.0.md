# Roadmap: v0.15.2 to v1.0.0

Full release path for modern-pdf-lib from current version (0.15.1) through production-ready 1.0.0.

---

## 0.15.x — JPEG Recompression + Image Optimization Pipeline (remaining patches)

0.15.0 and 0.15.1 are already released. These patches complete the image optimization story.

| # | Version | Update | Effort |
|---|---------|--------|--------|
| 0 | 0.15.2 | ICC color profile extraction and embedding during image recompression | M |
| 1 | 0.15.3 | Lanczos3 resampling algorithm for image downscaling | M |
| 2 | 0.15.4 | Pre-optimization analysis — report current vs optimal encoding per image | S |
| 3 | 0.15.5 | Filter images by page range, minimum size, color space, or name pattern | S |
| 4 | 0.15.6 | Detect and handle JPEG arithmetic coding (SOF9 marker) | S |
| 5 | 0.15.7 | Preserve EXIF orientation, DPI tags, and copyright metadata during recompression | S |
| 6 | 0.15.8 | Concurrent image processing with configurable parallelism | M |
| 7 | 0.15.9 | CLI progress bar, per-image breakdown, and summary statistics | S |
| 8 | 0.15.10 | Benchmark suite + compatibility tests against real-world image-heavy PDFs | S |

### Details

**0.15.2 — ICC Color Profile Preservation**
When recompressing images, ICC profiles (sRGB, Adobe RGB, ProPhoto) are currently stripped. Print workflows depend on accurate color profiles — without them, colors shift between screen and press.

**0.15.3 — Lanczos Resampling**
Current downscaling uses bilinear interpolation, which softens fine detail. Lanczos3 preserves sharpness in text-heavy scanned documents and architectural drawings — visible quality improvement at 150 DPI.

**0.15.4 — Compression Ratio Analysis**
Users want to preview savings before committing. "Image on page 3 is 2.4 MB PNG, would be 340 KB as JPEG Q80" lets users make informed decisions without trial-and-error.

**0.15.5 — Selective Optimization Filters**
"Optimize only pages 10-50" or "skip images under 50 KB" or "only touch RGB images." Real workflows need surgical control, not all-or-nothing.

**0.15.6 — JPEG Arithmetic Coding Detection**
Some PDFs contain arithmetic-coded JPEGs which most decoders reject. Detect these and transcode to Huffman coding during optimization so the output is universally compatible.

**0.15.7 — Image Metadata Preservation**
Recompressing strips EXIF data. Orientation tags prevent rotated photos; DPI tags affect print sizing; copyright metadata has legal implications. Preserve what matters, strip what doesn't.

**0.15.8 — Parallel Image Optimization**
A 200-page scanned PDF has 200 images. Sequential processing takes minutes. With concurrency: 4, the WASM JPEG encoder processes 4 images simultaneously — 3-4x faster on multi-core machines.

**0.15.9 — CLI Progress and Reporting**
`npx modern-pdf optimize` on a large file currently runs silently. Users need feedback: "Processing image 47/203... 62% complete. Saved 14.2 MB so far." Professional CLI UX.

**0.15.10 — Image Optimization Test Suite**
Test against scanned legal documents, photo books, medical reports, and architectural plans. Measure compression ratios, quality metrics (SSIM), and processing speed. Catch regressions.

## 0.16.0 — Browser Bundle + CDN + Web Worker + Playground

The library works in browsers via bundlers, but has no pre-built browser bundle, no CDN presence, and no Web Worker API. A 1.0 library must be usable without a build step.

| # | Version | Update | Effort |
|---|---------|--------|--------|
| 0 | 0.16.0 | Pre-built browser ESM bundle with tree-shaking side-effect annotations | L |
| 1 | 0.16.1 | CDN distribution — unpkg, jsdelivr, esm.sh with versioned URLs | S |
| 2 | 0.16.2 | Web Worker API — offload PDF generation off the main thread | L |
| 3 | 0.16.3 | IIFE/UMD bundle for legacy script-tag usage | M |
| 4 | 0.16.4 | Interactive playground — browser-based PDF editor with live preview | L |
| 5 | 0.16.5 | Bundle size analysis and code splitting for partial imports | M |
| 6 | 0.16.6 | Service Worker support for offline PDF generation | S |
| 7 | 0.16.7 | Content Security Policy (CSP) compatibility — nonce and integrity support | S |
| 8 | 0.16.8 | Browser-optimized WASM loader with fetch + streaming compilation | M |
| 9 | 0.16.9 | One-click download helper — ReadableStream to Blob to download link | S |
| 10 | 0.16.10 | Browser integration guide with React, Vue, Svelte, and vanilla JS examples | S |

### Details

**0.16.0 — Browser ESM Bundle**
The foundation. Without a dedicated browser build, users must configure bundlers to handle the full Node-targeting package. A clean browser ESM entry point enables everything else in this series.

**0.16.1 — CDN Distribution**
`<script type="module" src="https://cdn.jsdelivr.net/npm/modern-pdf-lib/dist/browser.mjs">` — zero install, zero config. Codepen, JSFiddle, quick prototypes need this.

**0.16.2 — Web Worker API**
Generating a 100-page PDF blocks the UI for seconds. A Worker wrapper lets users call `const bytes = await pdfWorker.save(doc)` without jank. Non-negotiable for production web apps.

**0.16.3 — IIFE/UMD Bundle**
Not everyone uses ESM. WordPress plugins, Shopify themes, and legacy apps need `<script src="modern-pdf-lib.min.js">` with a global `window.ModernPdf`. Meet users where they are.

**0.16.4 — Interactive Playground**
"Try before you install." A playground where users type code and see the PDF render instantly is the single best conversion tool. Every successful library has one.

**0.16.5 — Code Splitting**
The full library is 628 KB. Users who only create simple PDFs shouldn't pay for the parser, crypto, and form modules. Split into `modern-pdf-lib/create`, `/parse`, `/forms` sub-paths.

**0.16.6 — Service Worker Support**
PWAs and offline-first apps need to generate PDFs without network access. Ensure the library initializes correctly inside a Service Worker context (no DOM, no window).

**0.16.7 — CSP Compatibility**
Strict CSP environments block inline scripts and require integrity hashes. WASM instantiation via `WebAssembly.compile()` needs `wasm-unsafe-eval`. Document the minimum CSP and provide workarounds.

**0.16.8 — Browser WASM Loader**
`WebAssembly.compileStreaming(fetch(url))` is 2-3x faster than loading bytes then compiling. Auto-detect browser vs Node and use the optimal loading strategy for each WASM module.

**0.16.9 — Download Helper**
Every browser user needs `doc.save()` to file download. Provide `saveAsDownload(doc, 'invoice.pdf')` and `saveAsBlob(doc)` so users don't write the same Blob/URL boilerplate in every project.

**0.16.10 — Browser Integration Guide**
Framework-specific gotchas (React strict mode double-init, Vue reactivity on Uint8Array, Svelte SSR). Tested code samples for the top 3 frameworks plus vanilla JS.

## 0.17.0 — PDF/A Full Enforcement + veraPDF Validation

PDF/A compliance is claimed but never validated against the industry-standard veraPDF tool. At 1.0, it must pass real validation — not just internal checks.

| # | Version | Update | Effort |
|---|---------|--------|--------|
| 0 | 0.17.0 | veraPDF CI integration — automated validation on every test run | L |
| 1 | 0.17.1 | PDF/A-1b output intent and sRGB color space enforcement | M |
| 2 | 0.17.2 | Font embedding completeness — ToUnicode CMap for all embedded fonts | M |
| 3 | 0.17.3 | Transparency flattening for PDF/A-1 output | L |
| 4 | 0.17.4 | XMP metadata schema validation — required fields and namespaces | S |
| 5 | 0.17.5 | PDF/A-2b extensions — JPEG2000 images and transparency allowed | M |
| 6 | 0.17.6 | PDF/A-3 embedded file relationship (AF key) support | M |
| 7 | 0.17.7 | Prohibited feature enforcement — JavaScript, encryption, audio/video removal | M |
| 8 | 0.17.8 | PDF/A identification in XMP — pdfaid:part and pdfaid:conformance | S |
| 9 | 0.17.9 | enforcePdfA() post-processing — full cleanup pass on loaded documents | L |
| 10 | 0.17.10 | PDF/A compliance guide with veraPDF setup and ZUGFeRD e-invoicing example | S |

### Details

**0.17.0 — veraPDF CI Integration**
The foundation. Without veraPDF in CI, PDF/A compliance is a guess. Run every generated PDF/A test fixture through veraPDF and fail the build on violations.

**0.17.1 — Output Intent + sRGB**
PDF/A-1b requires an OutputIntent dictionary with an ICC profile. Currently missing — veraPDF fails immediately. Embed the sRGB ICC profile and set /GTS_PDFA1 as the output intent subtype.

**0.17.2 — ToUnicode CMap Completeness**
PDF/A requires every glyph to map to Unicode. Standard 14 fonts skip ToUnicode because PDF viewers infer it — veraPDF doesn't. Generate CMap tables for all fonts, including the standard set.

**0.17.3 — Transparency Flattening**
PDF/A-1 (based on PDF 1.4) prohibits transparency. If a user draws with opacity < 1.0 and then calls enforcePdfA('1b'), the transparency must be flattened to opaque equivalents.

**0.17.4 — XMP Schema Validation**
PDF/A requires specific XMP properties: dc:title, xmp:CreateDate, pdf:Producer, pdfaid:part, pdfaid:conformance. Validate that all mandatory fields are present and correctly namespaced.

**0.17.5 — PDF/A-2b Extensions**
PDF/A-2 relaxes PDF/A-1 restrictions. Allow transparency and JPEG2000 when targeting level 2, but still enforce all other requirements. Separate validation profiles per level.

**0.17.6 — PDF/A-3 AF Key**
PDF/A-3 allows embedded files but requires /AF (associated files) arrays and /AFRelationship values. Invoice standards like ZUGFeRD/Factur-X embed XML inside PDF/A-3 — this key unlocks e-invoicing.

**0.17.7 — Prohibited Feature Removal**
PDF/A forbids JavaScript actions, encryption, embedded audio/video, and launch actions. enforcePdfA() must actively strip these when present in loaded PDFs, not just skip adding them.

**0.17.8 — PDF/A Identification**
The PDF/A identification schema in XMP is how viewers know a file claims PDF/A compliance. Without pdfaid:part=1, pdfaid:conformance=B, Acrobat won't show the blue PDF/A bar.

**0.17.9 — enforcePdfA() Full Cleanup**
Loading a non-compliant PDF and calling enforcePdfA() must fix everything: add missing metadata, strip prohibited features, embed missing fonts, add output intents.

**0.17.10 — PDF/A Guide**
Full workflow: create PDF, enforce compliance, validate with veraPDF, archive. Include a ZUGFeRD invoice example embedding XML in PDF/A-3b.

## 0.18.0 — QR Codes + Barcodes (7 Formats)

PDF invoices need QR codes. Shipping labels need Code 128. Product packaging needs EAN-13. Every barcode is currently "bring your own image" — native generation removes a dependency.

| # | Version | Update | Effort |
|---|---------|--------|--------|
| 0 | 0.18.0 | QR code generator — native QR code rendering directly in PDF | L |
| 1 | 0.18.1 | Code 128 barcode — the universal shipping/logistics barcode | M |
| 2 | 0.18.2 | EAN-13 and EAN-8 barcodes — retail product identification | M |
| 3 | 0.18.3 | UPC-A barcode — North American retail standard | S |
| 4 | 0.18.4 | Code 39 barcode — government, military, and healthcare standard | S |
| 5 | 0.18.5 | ITF (Interleaved 2 of 5) barcode — shipping cartons and distribution | S |
| 6 | 0.18.6 | PDF417 2D barcode — IDs, boarding passes, and government documents | L |
| 7 | 0.18.7 | Data Matrix 2D barcode — industrial and pharmaceutical marking | L |
| 8 | 0.18.8 | Barcode styling — colors, quiet zones, human-readable text positioning | S |
| 9 | 0.18.9 | Barcode reading from existing PDFs — extract and decode barcodes | L |
| 10 | 0.18.10 | Barcode generation guide with format comparison and industry use cases | S |

### Details

**0.18.0 — QR Code Generator**
QR codes appear on every invoice, ticket, boarding pass, and payment slip. Native generation means no external library, no rasterization — crisp vector output at any zoom level.

**0.18.1 — Code 128**
Code 128 is the standard for shipping labels, inventory, and warehouse management. FedEx, UPS, and Amazon all use it. Highest density of the linear barcodes — encodes full ASCII.

**0.18.2 — EAN-13 / EAN-8**
Every product sold in a store has an EAN barcode. Generating product labels, price tags, and catalog PDFs requires EAN-13 (full) and EAN-8 (compact) with correct check digit calculation.

**0.18.3 — UPC-A**
UPC-A is the 12-digit barcode on every product in the US and Canada. While EAN-13 is a superset, many systems specifically require UPC-A formatting with its distinctive guard patterns.

**0.18.4 — Code 39**
Code 39 is mandated by the US Department of Defense (MIL-STD-1189) and used across healthcare (HIBC). Self-checking with no mandatory check digit — simple and universally scannable.

**0.18.5 — ITF**
ITF-14 is the standard for outer carton marking in logistics. Higher density than Code 39 for numeric-only data. Required for GS1-128 shipping container labels.

**0.18.6 — PDF417**
PDF417 is used on US driver's licenses, airline boarding passes, and government forms. High data capacity (up to 1,850 characters) with built-in error correction. Required by IATA and AAMVA standards.

**0.18.7 — Data Matrix**
Data Matrix is required for pharmaceutical serialization (EU FMD, US DSCSA) and electronics component marking. Smallest footprint of any 2D code — reads reliably at 2mm x 2mm.

**0.18.8 — Barcode Styling**
Barcodes need configurable bar color, background, quiet zone width, and human-readable text (above/below/none). Brand guidelines require specific colors. Standards mandate minimum quiet zones.

**0.18.9 — Barcode Reading**
Parse a shipping label PDF and extract the tracking number. Read QR codes from scanned invoices. Decode barcodes from rasterized images embedded in PDFs. Closes the read/write loop.

**0.18.10 — Barcode Guide**
"Which barcode do I use?" — decision matrix comparing all 7+ formats by data capacity, size, industry standard, and scanner compatibility. Code examples for invoices, shipping, and product labels.

## 0.19.0 — Table Layout Engine + Pagination + Auto-Width

The most requested feature in PDF libraries. pdf-lib doesn't have it. This is the differentiator.

| # | Version | Update | Effort |
|---|---------|--------|--------|
| 0 | 0.19.0 | Basic table rendering — rows, columns, cell text, and borders | L |
| 1 | 0.19.1 | Column width modes — fixed, auto-fit, percentage, and weighted | M |
| 2 | 0.19.2 | Cell padding, margins, and alignment (horizontal + vertical) | M |
| 3 | 0.19.3 | Row spanning (rowspan) and column spanning (colspan) | M |
| 4 | 0.19.4 | Automatic page breaks with header row repetition | L |
| 5 | 0.19.5 | Cell background colors and alternating row stripes | S |
| 6 | 0.19.6 | Nested tables — table inside a table cell | L |
| 7 | 0.19.7 | Rich cell content — images, styled text runs, and multi-line wrapping | M |
| 8 | 0.19.8 | Table styling presets and border customization API | M |
| 9 | 0.19.9 | Text overflow handling — wrapping, truncation, and ellipsis | M |
| 10 | 0.19.10 | Table layout guide with invoice, report, and schedule examples | S |

### Details

**0.19.0 — Basic Table Rendering**
The foundation. `page.drawTable({ rows, columns, x, y })` with simple text cells and single-line borders. Every invoice, report, and receipt needs a table. This unblocks the #1 feature request.

**0.19.1 — Column Width Modes**
"Name column takes 40%, ID takes 100px, Description fills remaining space." Without flexible column sizing, every table requires manual pixel math. Auto-fit measures content and picks optimal widths.

**0.19.2 — Cell Padding and Alignment**
Text crammed against cell borders looks unprofessional. Padding (inner spacing), margins (outer spacing), and alignment (left/center/right, top/middle/bottom) are the minimum for presentable output.

**0.19.3 — Row/Column Spanning**
Invoice headers that span all columns. "Ship To" cells that span 3 rows. Without spanning, users fake it with invisible borders and manual positioning — fragile and error-prone.

**0.19.4 — Automatic Page Breaks**
A 200-row table must split across pages automatically. When it breaks, repeat the header row on each new page so readers always know which column is which. The killer feature of a table engine.

**0.19.5 — Cell Backgrounds and Stripes**
Zebra-striped rows improve readability. Header rows need distinct backgrounds. Conditional cell coloring (red for negative values, green for positive) is expected in financial reports.

**0.19.6 — Nested Tables**
Complex layouts need sub-tables: a product table inside an order table, a schedule grid inside a calendar cell. Recursive layout calculation with correct height propagation.

**0.19.7 — Rich Cell Content**
Cells need more than plain text. Product images in catalog tables, bold headers mixed with normal text, and long descriptions that word-wrap within the cell boundary.

**0.19.8 — Table Styling Presets**
`style: 'minimal'`, `'striped'`, `'bordered'`, `'professional'` — one-line table themes. Plus fine-grained control: per-cell borders, border colors, dashed/solid/double line styles.

**0.19.9 — Text Overflow Handling**
When text doesn't fit a cell: wrap to multiple lines (grow row height), truncate with "..." ellipsis, or shrink font size to fit. Each strategy suits different use cases (reports vs labels vs forms).

**0.19.10 — Table Layout Guide**
Three complete examples: a professional invoice with totals row, a financial report with nested sub-tables and conditional formatting, and a weekly schedule with merged time slots. Copy-paste ready.
